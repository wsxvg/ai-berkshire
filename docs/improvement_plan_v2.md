## 零、核心立场

**AI Berkshire 的本质是一个信号发现系统，不是一个交易系统。**

我们的价值主张是：通过跟踪大佬持仓，发现被市场忽视的共识信号，帮助用户做出更好的基金投资决策。

**核心假设**：多位高收益大佬同时买入同一只基金 → 这只基金未来表现大概率优于市场。

**这个假设必须先验证，否则后续所有改进都是空中楼阁。**

### Pivot 方案（如果信号无效）

如果 M1-M2 验证发现共识信号的超额收益 < 5% 或胜率 < 55%，项目转向：

**方向：从"跟大佬买"转向"数据驱动的基金推荐"**

- 保留数据采集能力（持仓、交易流水、排名）
- **保留投资建议能力**，但改变信号来源：
  - 不再依赖"大佬共识"作为唯一信号
  - 转为综合多维度数据：基金历史业绩 + 费率 + 基金经理能力 + 市场环境
  - AI 分析这些数据后给出买入/卖出/持有建议
- 用户价值不变：**你仍然告诉我们买什么卖什么**，只是依据从"大佬操作"变成"数据+AI分析"

## 二、P0：信号有效性验证（前瞻性跟踪，非回测）

### 为什么不用回测

**回测对本项目无意义，原因：**

1. **大佬用 AI 做决策**：我们跟踪的大佬本身就在用带联网功能的 AI 工具（类似本项目），他们的决策基于实时数据。用历史快照回测等于用"静态照片"验证"动态行为"，结果必然失真。
2. **信息不对称**：大佬可能基于我们无法获取的实时信息（舆情、消息面）做决策，回测假设所有人信息对等，但现实不是。
3. **幸存者偏差**：我们只跟踪了当前名单，但全平台 Top10 在变。历史回测用的是"现在的名单"，不代表过去的信号质量。

### 正确的验证方式：前瞻性跟踪

**核心思路**：不回测历史，而是记录未来的信号表现，持续积累真实数据。

**数据需求**：
- 每日持仓快照（已有，从 pipeline 归档文件获取）
- 基金净值数据（需接入天天基金 API）

**回测规则**：
```
信号触发：T 日 N 位大佬同时买入基金 X
执行策略：T+1 日开盘买入，持有 K 天后卖出
对比基准：同期沪深300指数
评估指标：年化收益率、夏普比率、最大回撤、胜率
```

**参数扫描**：
- N = 2, 3, 5（共识人数阈值）
- K = 7, 30, 90, 180（持有天数）
- 评估不同参数组合的表现
- 观察信号衰减：T+7、T+30、T+90 的共识人数变化

### 信号记录数据结构

```json
{
  "signal_id": "2026-06-30-012922",
  "date": "2026-06-30",
  "fund_code": "012922",
  "signal_type": "cross_consensus",
  "consensus_strength": 5,
  "entry_nav": 4.9191,
  "market_env": "bull|bear|sideways",
  "fund_category": "QDII|equity|bond",
  "nav_7d": null,
  "nav_30d": null,
  "nav_90d": null,
  "signal_strength_decay": null,
  "failure_reason": null,
  "status": "pending"
}
```

**归因维度**（结构化，非 LLM）：
- `market_env`：市场环境（牛市/熊市/震荡）
- `fund_category`：基金类型（QDII/权益/债券）
- `consensus_strength`：共识人数
- `failure_reason`：信号失败原因（可从数据推断）

### 技术实现

```python
# 新增文件：scripts/signal_tracker.py
# T 日记录信号 → 每日回填净值 → 统计胜率和归因
```

### 自动归类规则

- `market_env`：基于沪深300 最近 20 日涨跌幅自动判断
  - 涨幅 > 5%：bull
  - 跌幅 > 5%：bear
  - 其他：sideways
- `fund_category`：基于天天基金分类接口自动标注（QDII/权益/债券/货币）

## 三、P0：多数据源聚合

### 为什么这是 P0

单数据源（京东金融 API）是系统性风险：
- API 变更/下线 → 系统瘫痪
- 数据不完整（缺净值历史、基金经理详情） → 信号质量差
- 无法做回测验证 → 核心假设无法验证

### 数据源矩阵

| 数据类型 | 当前 | 需要补充 | 来源 | 优先级 |
|----------|------|---------|------|--------|
| 用户持仓 | ✅ 京东金融 | — | — | — |
| 交易流水 | ✅ 京东金融 | — | — | — |
| 基金净值历史 | ❌ | ✅ 必须 | 天天基金 API / AkShare | P0 |
| 基金费率结构 | ❌ | ✅ 必须 | 天天基金 | P0 |
| QDII限购状态 | ❌ | ✅ 必须 | 京东金融（已有接口）/ 天天基金 | P0 |
| 基金规模 | ❌ | ✅ 高优 | 天天基金 / 晨星 | P0 |
| 基金经理详情 | ❌ | ✅ 中优 | 天天基金 | P1 |
| 基金公告 | ❌ | ✅ 中优 | 京东金融（已有接口） | P1 |
| 基金评级 | ❌ | ✅ 中优 | 晨星 / 天天基金 | P1 |
| 市场行情 | ❌ | ✅ 低优 | AkShare / YFinance | P2 |

> **费率和限购是信号落地的关键约束**：大佬买入的基金如果限购或费率过高，用户根本买不了。

### 优先补充

1. **天天基金 API**：净值历史 + 基金规模 + 基金经理
2. **AkShare**：备选数据源，防止京东金融 API 变更
3. **历史快照归档**：每日自动归档 holdings_snapshot，为回测做准备

## 四、P1：架构模式借鉴（非功能搬运）

### 4.1 借鉴 TradingAgents 的 StateGraph 模式

**当前问题**：pipeline 是线性脚本，某一步失败就整体失败。

**借鉴方案**：用状态机管理 pipeline 各阶段

```
PIPELINE_STATES:
  AUTH → HOLDINGS → TRADING → RANKING → SIGNALS → REPORT
  
每个状态有：
  - entry: 进入该状态时执行
  - exit: 离开该状态时执行  
  - on_error: 失败时的恢复策略
  - dependencies: 前置状态列表
```

**好处**：
- 某个 API 失败不影响其他阶段
- 可以从断点恢复
- 每个阶段独立超时和重试

### 4.2 借鉴 QuantDinger 的双策略范式

**当前问题**：信号计算是硬编码的规则，难以扩展。

**借鉴方案**：分离"规则引擎"和"灵活脚本"

```
RuleEngine: 
  - 当前的信号计算逻辑（强共识、买入护盾等）
  - 确定性输出，不需要 LLM
  - 适用于大多数场景

ScriptEngine:
  - 用户自定义的分析脚本
  - 可以调用 LLM（可选）
  - 适用于特殊场景（如"分析某基金为什么被多人买入"）
```

**好处**：
- 规则引擎保证基础功能稳定
- 脚本引擎支持灵活扩展
- LLM 是可选插件，不影响零成本定位

### 4.3 借鉴 daily_stock_analysis 的推送模式

**当前问题**：用户需要主动打开 GitHub 看报告。

**借鉴方案**：报告生成后自动推送到用户选择的渠道

```
推送渠道：
  - 飞书机器人（ webhook ）
  - 企业微信机器人（ webhook ）
  - 邮件（ SMTP ）
  - Telegram Bot

推送内容：
  - 每日信号摘要（1分钟看完）
  - 完整报告链接（想看详情点进去）
```

## 五、P1：反思学习机制

### 为什么需要

当前信号是"一次性"的——今天推送了买入信号，但不知道这个信号最终是对是错。没有反馈循环，信号质量无法提升。

### 设计方案

```
数据结构：
{
  "signal_id": "2026-06-30-fund-012922",
  "date": "2026-06-30",
  "fund_code": "012922",
  "fund_name": "易方达全球成长精选",
  "signal_type": "cross_consensus",
  "users_followed": ["家庭的温暖", "Z先生养基"],
  "users_ranking": ["简单生活快乐买基"],
  "entry_price": 4.9191,  # 买入时净值
  "exit_price": null,      # 卖出时净值（30天后回填）
  "return_30d": null,      # 30天收益率（回填）
  "win": null,             # 是否跑赢基准（回填）
  "reflection": null       # LLM 反思（可选）
}
```

**工作流程**：
1. T 日：记录信号 + 买入价格
2. T+30 日：回填卖出价格 + 计算收益率
3. T+31 日：判断信号对错，生成反思
4. 下次类似信号出现时：注入历史反思

### 技术实现

需要补充的数据：
- **基金净值历史**（天天基金 API）——计算任意区间的收益率
- **信号记录表**（SQLite）——存储每次信号的完整上下文

## 六、AI 集成策略（已内建，无需额外 API）

### 核心认知

**本项目已经运行在 AI IDE（Claude Code / OpenCode）上。AI 本身就是 LLM，不需要额外调 API。**

### 核心 Skill：fund-trade（今日交易建议）

**使用场景**：交易日下午3点前，用户说"今天买什么"，AI 读取 pipeline 数据给出建议。

**工作流**：
```
用户："今天买什么"
  ↓
AI 读取 data/auto/status.json（信号汇总）
  ↓
AI 读取 data/holdings_snapshot.json（大佬持仓）
  ↓
AI 读取 data/trading_records_cache.json（今日交易）
  ↓
AI 综合分析：买入信号 + 卖出信号 + 持仓建议 + 风险提示
  ↓
AI 输出：具体建议（买什么、卖什么、持有、理由、风险）
```

**输出示例**：
```
🟢 建议买入：易方达全球成长精选C（7位大佬持有，排名大佬也买入）
🔴 建议减仓：XX基金C（有人清仓，无买入信号）
⚪ 建议持有：YY基金A（多人持有，无卖出信号）
⚠️ 风险提示：QDII近期回撤大，注意汇率风险
```

### 为什么这样设计

1. **简单**：不需要复杂的状态机或反思循环
2. **实用**：直接解决"今天怎么操作"的问题
3. **已有数据**：pipeline 已经提供了所有需要的数据
4. **AI 内建**：不需要额外 API，AI 在 skill 框架内运行
