# AI Berkshire 场外基金分析能力改进规划

> 基于对 ai-berkshire 项目架构的深入分析，结合京东金融42个已验证API，提出的场外基金分析能力扩展方案。

---

## 一、项目现状深度分析

### 1.1 架构本质

ai-berkshire 不是一个应用，而是一套 **Claude Code Skills 工作流引擎**：

```
用户输入 → Skill(.md)定义执行步骤 → Claude Agent调用工具(Python) → 输出结构化报告
```

每个 Skill 是一个 `.md` 文件，定义了：
- 输入格式（`$ARGUMENTS`）
- 分步骤执行流程（第一步、第二步...）
- 数据源优先级（financial-data.md）
- 输出格式（reports/目录结构）

每个 Tool 是一个 Python 脚本，遵循 **零外部依赖** 原则（只用stdlib），职责明确：
- `financial_rigor.py` — 精确计算（Decimal）
- `report_audit.py` — 数据抽检（15%抽样验证）
- `stock_screener.py` — 股票筛选
- `xueqiu_scraper.py` — 数据抓取

### 1.2 现有能力边界

| 能力 | 现有Skill | 分析对象 |
|------|-----------|---------|
| 个股深度分析 | `/investment-research` | 单家公司 |
| 多视角对抗 | `/investment-team` | 单家公司（4个Agent并行） |
| 买入前筛选 | `/investment-checklist` | 单家公司（六关检查） |
| 行业全景 | `/industry-research` | 整个产业链 |
| 行业漏斗 | `/industry-funnel` | 从全市场筛到3只 |
| 财报精读 | `/earnings-review` | 单家公司财报 |
| 买入后跟踪 | `/thesis-tracker` | 单家公司的投资论文 |
| 组合管理 | `/portfolio-review` | 已有持仓的再平衡 |
| 数据验证 | `report_audit.py` | 报告中的财务数据15%抽样 |

**核心发现：所有现有能力都围绕"股票/公司"，没有任何基金相关能力。**

### 1.3 关键约束（从项目CLAUDE.md提取）

- 所有分析必须基于事实和数据，严禁主观臆断
- 关键数据至少2个来源交叉验证
- PE/ROE等用 `tools/financial_rigor.py` 精确计算
- 报告语言中文，风格直接犀利
- 输出到 `reports/` 目录，按公司名建文件夹

## 二、场外基金投资的特殊性

### 2.1 与股票投资的本质差异

| 维度 | 股票投资（现有能力） | 场外基金投资（待建能力） |
|------|---------------------|------------------------|
| 分析对象 | 公司基本面 | 基金质量 + 底层资产 + 交易成本 |
| 决策变量 | 估值、护城河、管理层 | T+N、费率、限额、基金经理风格 |
| 时间成本 | 随时买卖 | T+1~T+10确认，15:00截止 |
| 持有成本 | 无（除佣金） | 管理费+托管费+销售服务费（每日计提） |
| 退出成本 | 佣金 | 赎回费分档（7天内1.5%罚款） |
| 信息来源 | 财报、公告 | 基金季报、京东金融API |

### 2.2 场外基金的"隐藏成本"（全部从API获取）

```
getFundTradeRulesPageInfo 返回的完整规则包：
├── 申购费率（分档，含平台折扣）
├── 赎回费率（持有天数分档）
│   ├── < 7天：1.5%（证监会强制）
│   ├── 7天~365天：0.XX%（每只基金不同）
│   └── ≥ 365天：0% 或极低
├── 管理费：X.XX%/年（每日从NAV扣除）
├── 托管费：X.XX%/年
├── 销售服务费：X.XX%/年（C类份额）
├── T+N确认天数（每只基金不同）
├── 买入截止时间（15:00前/后）
└── 日累计限额
```

**关键认知：这些数值每只基金都不同，必须逐基金从API获取，绝不能硬编码。**

### 2.3 大佬跟踪的特殊价值

场外基金的持仓数据是**季度公开**的（基金季报），但京东金融的实盘功能提供了**实时可见**的持仓和交易记录。这意味着：
- 季报数据：延迟3个月，但覆盖全部持仓
- 实盘数据：实时可见，但只覆盖公开持仓的大佬
- 两者结合 = 完整的画面

## 三、改进方案：4个新Skill + 1个新Tool

### 3.1 新增 Tool: `tools/jd_finance_api.py`

**设计原则**：遵循项目现有的零外部依赖模式（只用stdlib），封装京东金融API调用。

```python
# 核心功能
class JDFinanceAPI:
    def get_user_holdings(target_uid, fund_type='fund') -> list[dict]
    def get_trading_records(target_uid, page=1, size=20) -> list[dict]
    def get_fund_trade_rules(fund_code) -> dict  # T+N、费率、限额
    def get_fund_holdings_distribution(fund_code, quarter=None) -> dict
    def get_fund_manager(manager_id) -> dict
    def get_followed_users() -> list[dict]  # 从圈子动态提取uid
    
    # 所有返回值都包含完整的交易规则字段，不硬编码任何数值
```

**与现有工具的关系**：
- `financial_rigor.py` 负责精确计算 → jd_finance_api.py 负责数据获取
- `report_audit.py` 负责数据验证 → 可扩展支持基金数据验证
- `xueqiu_scraper.py` 负责股票数据抓取 → jd_finance_api.py 负责基金数据获取

### 3.2 新增 Skill: `skills/fund-checklist.md`

**对标**：`/investment-checklist`（股票六关筛选）

**改造**：在股票Checklist的六关基础上，增加基金特有的约束维度：

```
第一关：我能理解这只基金吗（能力圈）
  → 基金类型、投资策略、跟踪标的
  → 数据来源：getFundDetailProfilePageInfo

第二关：好基金吗（质量）
  → 夏普比率、最大回撤、同类排名
  → 数据来源：getFundHistoryPerformancePageInfo

第三关：基金经理靠谱吗（管理层）
  → 任职年限、管理规模、历史业绩、获奖
  → 数据来源：getFundManagerDetailPageInfo + debtBaseManagerDetailIndex

第四关：费率合理吗（成本）← 股票没有这一关
  → 从 getFundTradeRulesPageInfo 实时获取：
    申购费率分档 + 赎回费率分档 + 管理费 + 托管费
  → 计算持有1年/3年/5年的实际成本
  → C类份额 vs A类份额的费率对比

第五关：流动性够吗（退出成本）← 股票没有这一关
  → T+N确认天数（从API获取）
  → 买入截止时间
  → 日累计限额
  → 基金规模（太小有清盘风险）

第六关：大佬在买吗（聪明钱验证）
  → queryUserFundHoldingInfo 获取关注人持仓
  → getPageMutilData 获取交易记录
  → 交叉验证信号强度
```

### 3.3 新增 Skill: `skills/fund-monitor.md`

**核心功能**：大佬持仓变化监控

```
输入：无（自动拉取11位关注人）
执行流程：
  1. 调用 jd_finance_api.get_followed_users() 获取关注人列表
  2. 逐人调用 get_user_holdings() 获取最新持仓
  3. 与 data/fund_snapshots/ 中的上次快照对比
  4. 输出变化报告：
     - 新增持仓（大佬新买了什么）→ 自动触发 /fund-checklist
     - 减持（大佬在卖什么）→ 标记风险
     - 清仓（全卖了）→ 重点关注
  5. 识别"共识信号"：多人同时买入同一只基金
  6. 保存本次快照到 data/fund_snapshots/
```

### 3.4 新增 Skill: `skills/fund-penetration.md`

**核心功能**：基金持仓穿透 → 用现有框架分析底层资产

```
输入：基金代码
执行流程：
  1. getFundInvestmentDistributionPageInfo → 提取重仓股
  2. 对每只重仓股 → 调用 /investment-checklist
  3. 输出：基金质量 = Σ(重仓股权重 × 个股评分) × 基金经理能力系数

基金经理能力系数计算（从API实时获取）：
  ├─ 数据源：getFundManagerDetailPageInfo → 五维雷达评分
  ├─ 系数 = (收益能力×0.3 + 选股能力×0.3 + 避险能力×0.2 + 机会把握×0.2) / 5
  ├─ 任职年限修正：>5年 → ×1.1，<2年 → ×0.8
  └─ 最终系数范围：0.5 ~ 1.5（归一化到基准1.0）
```

### 3.5 新增 Skill: `skills/fund-quarterly.md`

**核心功能**：历史持仓季度变化追踪

```
输入：基金代码 或 "all"
执行流程：
  1. getFundInvestmentDistributionPageInfo（遍历6个季度）
  2. 对比相邻季度：新增/清仓/比例变化
  3. 识别风格漂移
  4. 输出季度调仓报告
```

## 六、认证机制设计（评委建议补充）

### 6.1 认证架构

京东金融API使用Cookie认证，涉及两个层面：

| 层面 | 机制 | 存储位置 |
|------|------|---------|
| 浏览器登录态 | `pt_key` + `pt_pin` + `pt_token` | `data/jd_auth/cookies.json` |
| 平台会话 | `s_key` + `s_pin` + `guid` | 同上 |

### 6.2 生命周期管理（支持自动登录）

```
启动 → 读取 cookies.json → 尝试调用 pcQueryUserInfo
  ├─ 成功 → 使用该session
  └─ 失败(resultCode=3) → 自动触发Playwright登录流程
       → 打开京东金融登录页
       → 用户扫码/输密码（仅首次或Cookie过期时）
       → Playwright自动抓取Cookie → 更新 cookies.json
       → 重试API调用
```

### 6.3 Claude Code环境下的Cookie更新流程

```python
# tools/jd_auth_helper.py — 辅助脚本
# 方式A：Playwright自动抓取（推荐）
def auto_refresh_cookie():
    """启动Playwright → 打开京东金融登录页 → 等用户扫码 → 抓取Cookie写入cookies.json"""
    # Playwright打开 btfront.jd.com → 检测登录态 → 未登录则跳转登录页
    # 用户完成登录后，从浏览器上下文中提取所有jd.com域的Cookie
    # 写入 data/jd_auth/cookies.json

# 方式B：环境变量注入（备选）
# JD_COOKIE="pt_key=xxx;pt_pin=xxx;..." python tools/jd_finance_api.py
```

**Claude Code集成流程**：
1. `jd_finance_api.py` 启动时读取 `data/jd_auth/cookies.json`
2. 调用 `pcQueryUserInfo` 探测登录态
3. 失败 → 调用 `tools/jd_auth_helper.py auto` → Playwright自动抓取
4. 用户仅需在弹出的浏览器中扫码一次
5. 后续自动使用新Cookie

## 八、缓存与性能策略（评委建议补充）

### 8.1 缓存架构

```
data/fund_cache/
├── holdings_{uid}_{date}.json     — 持仓快照（按用户按日期）
├── trade_rules_{fund_code}.json   — 交易规则缓存（30天有效）
├── fund_detail_{fund_code}.json   — 基金详情缓存（7天有效）
└── penetration_{fund_code}.json   — 穿透分析缓存（季度有效）
```

### 8.2 性能预算

| 场景 | 无缓存 | 有缓存 |
|------|--------|--------|
| fund-monitor（11人×每人5只） | ~55次API调用 | ~10次（仅变化的） |
| fund-penetration（1只基金×10只重仓股） | ~15次API调用 | ~3次（仅新增/变化） |
| fund-quarterly（6个季度） | ~6次API调用 | ~1次（仅最新季度） |

### 8.3 缓存失效策略

| 缓存类型 | 有效期 | 失效触发器 |
|----------|--------|-----------|
| 持仓快照 | 当日 | fund-monitor每日运行时覆盖 |
| 交易规则 | 30天 | 超时自动清除 |
| 基金详情 | 7天 | 超时自动清除 |
| 穿透分析 | 当季 | **双触发**：①时间触发：每季度首月15日批量清理（季报披露截止日）；②数据触发：API返回的reportDate与缓存不一致时即时失效 |

## 十一、回滚与降级方案（评委建议补充）

### 核心原则
`jd_finance_api.py` 采用独立模块设计，与现有股票Skill链完全解耦。京东金融API失效时，18个现有股票Skill照常运行，不受影响。

### 降级模式
基金Skill支持 `--offline` 参数：
```
/fund-checklist 华安黄金ETF联接A --offline
```
**offline模式行为**：
- 跳过所有京东金融API调用
- 仅使用 `data/fund_cache/` 中的本地缓存数据
- 报告中标注"数据截至 {缓存日期}，未联网更新"
- 穿透分析仍可执行（基于缓存的持仓数据）

### 降级触发条件
| 场景 | 自动降级为 |
|------|-----------|
| Cookie失效且无法自动刷新 | offline模式 + 提示更新Cookie |
| API返回错误码 | 使用最近一次成功缓存 |
| 网络不可达 | offline模式 |
| API大规模变更（接口名改变） | jd_finance_api.py集中修复，其他模块不动 |
