## 二、改进方向一：穿透持仓估值（P0 🔥）

### 目标

对基金的前十大重仓股，获取 PE/PB/市值数据，计算加权估值分位，判断基金"贵不贵"。

### 数据流

```
                   getQuoteExtendUseUniqueCodeWithCache
                   ┌────────────────────────────────────┐
  --fund-holdings  │  {                                │
  返回 top_stocks  │    uniqueCode: "SH-688041",       │
  [{name, code,   │    peRatio: 298.33,      ← 市盈率   │
    ratio: "9.5%"  │    pnaRatio: 34.61,      ← 市净率   │
    change: "增持" │    peDynamic: 295.91,    ← 动态PE   │
  }]               │    netAssetPerShare: 10.11,        │
                   │    marketValue: 8132亿,             │
                   │    lastPrice: 349.9    ← 当前股价   │
                   │  }                                 │
                   └────────────────────────────────────┘
                                  ↓
                    ┌──────────────────────────┐
                    │  穿透估值评分 (0-5分)       │
                    │                          │
                    │  加权PE分位:               │
                    │    Σ(每只重仓股PE × 占比)    │
                    │    ÷ 指数历史PE分位         │
                    │                          │
                    │  加权PB分位: 同上           │
                    │                          │
                    │  行业集中度:                │
                    │    top3行业占比 > 70% → 风险 │
                    │                          │
                    │  最终分 = PE分×0.4          │
                    │        + PB分×0.3          │
                    │        + 集中度分×0.3       │
                    └──────────────────────────┘
                                  ↓
                   嵌入五维评分 → 成为第六维「估值分」
                   或者作为质量分的子指标（20%权重）
```

### 关键设计决策

**为什么能直接做？** `getQuoteExtendUseUniqueCodeWithCache` **不需要 cookie**，`fund_holdings` API **不需要 cookie**，所以穿透估值可以在缓存模式下运行。

**代码转换**：持仓股票代码（如 `688041`）→ 需要映射为 `uniqueCode` 格式（`SH-688041`）。美股代码如 `TSM`→`US-TSM`。港股如 `0700`→`HK-0700`。

**估值锚点**：A 股用沪深 300 历史 PE 分位做参照，美股用纳斯达克历史 PE 分位，港股用恒生历史 PE 分位。可以使用 `getSimpleQuoteUseUniqueCodes` 获取指数最新PE。

### 实施规模

- 新增 1 个 API 封装函数（`get_stock_quote_extended`）
- 新增 1 个估值评分函数（`score_penetration_valuation`）
- 修改 `fund_scorer.py` 的 `score_fund()`（增加第六维或质量分子指标）
- 修改 `generate_html.py`（报告中显示重仓股 PE 表）
- **预计 1~2 天**

## 三、改进方向二：评分历史追踪（P1）

### 目标

每日记录评分快照，在 HTML 报告中显示评分变化趋势线，识别基金是在变好还是变差。

### 数据流

```
每日管道运行
     ↓
评分引擎输出 FundScore
     ↓
写入 data/fund_scores_history.jsonl
(追加模式，每行一个时间戳+各基金评分)
     ↓
generate_html.py 读取历史
     ↓
在报告中展示趋势：
- 本次 vs 上次总分变化
- 变化最大的维度
- 连续 N 天上升/下降标记
```

### 存储格式

```jsonl
{"date": "2026-07-01", "scores": {
  "013841": {"total": 3.64, "quality": 4.3, "cost": 3.0, "manager": 4.2, "momentum": 4.3, "smart_money": 2.8},
  "012922": {"total": 3.24, "quality": 4.6, ...},
  ...
}}
```

### 报告展示

```
013841 银华集成电路混合C
总分: 3.64 (↑0.17)  📈 连续3日上升
 质量: 4.3 (→)
 成本: 3.0 (→)
 经理: 4.2 (→) 
 动量: 4.3 (↑0.15)  ← 主要变化来源
 聪明钱: 2.8 (→)
```

### `FundScore.compare()` 已有实现（无需重写）

当前 `FundScore` 类已经有 `compare(previous)` 方法，返回 `total_delta`、`dim_deltas`、`main_driver`、`override`。历史追踪只需要：

1. 写入历史 JSONL 文件（管道中加一行）
2. 读取上次评分（状态文件中存上次评分快照）
3. 报告显示变化

### 实施规模

- 改 `auto-pipeline.py`（追加一行写入历史文件）
- 改 `generate_html.py`（读取历史，显示趋势标记）
- **预计 0.5 天**

## 四、改进方向三：LLM 自动点评（P2）

### 目标

在 HTML 报告目前的固定文案基础上，增加一段由 LLM 生成的"今日解读"，用 2-3 句话概括今天的评分变化和市场信号。

### 设计

**原则**：LLM 只做**叙事解读**，不做评分计算。保持"评分是 Python 的事，LLM 是解读的事"的边界。

**数据输入**（传给 LLM）：
```json
{
  "date": "2026-07-01",
  "buy_signals": [{"name": "银华集成电路混合C", "score": 3.64, "buy_count": 5, "limit": "∞"}],
  "sell_signals": [{"name": "...", "score": 2.1, "sell_count": 2}],
  "watch": [{"name": "...", "score": 3.39}],
  "top_score_changes": [{"name": "...", "delta": 0.17, "driver": "momentum"}],
  "your_holdings": [{"name": "易方达全球成长C", "score": 3.24, "profit": "+1.7%", "big_player_buy": 0}]
}
```

**LLM 输出**：2-3 句短文本，嵌入 HTML 报告顶部：
```html
<p class="llm-summary">
今日评分最高的是银华集成电路混合C（3.64），但整体评分偏低——说明大佬共识集中在半导体赛道但短期动量过快，
追高风险较大。你持有的易方达全球成长C评分3.24偏低，主要是今天无人加仓（聪明钱分低）且QDII费率较高。
建议：银华集成电路可小额定投，易方达全球成长暂持观察。
</p>
```

**实现方式**：在 `generate_html.py` 末尾增加 LLM 调用。有三种选择：

| 方式 | 优点 | 缺点 | 推荐度 |
|------|------|------|:-----:|
| A. 直接调用 Claude API（anthropic-sdk） | 质量高，可控 | 需 API Key | ⭐⭐⭐ |
| B. 输出 JSON 到 stdout，让 Claude Code Skill 读取后补充 | 无需 API Key | 多一步操作 | ⭐⭐ |
| C. 用本地 LLM（ollama） | 免费 | 质量不如 Claude | ⭐ |

### 实施规模

- 新增 `tools/llm_commentary.py`（封装 LLM 调用逻辑）
- 改 `generate_html.py`（调用 LLM，嵌入输出）
- **预计 0.5 天（方式A）**

## 五、改进方向四：直接买什么（P1）

### 目标

基于评分+限额+大佬共识+自身持仓，计算今天**具体建议买入/卖出的基金和金额**。

### 决策逻辑

```
输入：五维评分 + 大佬交易信号 + 基金限额 + 自身持仓
                     ↓
          ┌──────────────────┐
          │  组合优化计算器    │
          │                  │
          │  约束条件：        │
          │    单只 ≤ 总资金15% │
          │    总仓位 ≤ 100%   │
          │    日限额 ≤ API限额 │
          │    QDII ≤ 总资金30% │
          │                   │
          │  目标：最大化 加权评分 │
          └──────────────────┘
                     ↓
          输出 "今日操作建议"
```

### 建议格式

```
🟢 建议买入（优先级排序）：
  1. 银华集成电路混合C (013841)
     评分 3.64 | 5人共识 | 无限购
     建议金额：¥5,000（定投）/ ¥10,000（一次性）
     
  2. 财通成长优选混合C (021528)
     评分 3.43 | 4人共识 | 日限¥100
     建议金额：¥100/天（日定投）

🟡 你持有的基金：
  1. 易方达全球成长C (012922)
     评分 3.24 | 今日无人加仓
     建议：持有不动，不加仓也不减仓
     
  2. 银华集成电路混合C (013841)  ← 你已持有¥100
     评分 3.64 | 5人共识
     建议：可加仓至¥5,000

🔴 建议卖出：
  1. xxx基金 (xxxxxx)
     理由：...
```

### 交互式资金分配器（HTML 输入框 + JS 计算）

**用户需求**：在 HTML 报告中输入手头的闲钱金额，系统自动按凯利公式等金融知识推荐具体投入金额。

**实现方式**：纯前端 JavaScript，所有数据已嵌入 HTML，无需后端计算。

**金融知识模型**：

| 概念 | 应用 | 计算方式 |
|------|------|---------|
| **凯利公式** | 单只基金的最佳仓位比例 | `f* = (p × b - q) / b`，其中 `p = score/5`（评分归一化为胜率），`b = 近1年收益率/最大回撤`（赔率），`q = 1-p` |
| **风险平价** | 多基金之间的资金分配 | 每只基金分配比例 ∝ 评分 × 凯利系数，但受限于风控上限 |
| **集中度限制** | 单只基金不超总资金15%，QDII不超30% | 硬约束，超过则自动削减 |
| **现金留存** | 预留20%现金应对赎回/加仓机会 | 建议投入 = 闲钱 × 80% |
| **DCA 定投** | 对有日限额的基金建议分批 | 一次性金额 > 日限额 → 改为日定投建议 |
| **持仓重叠惩罚** | 已持有的基金降低加仓优先级 | 已持有基金 → 建议金额 = 目标仓位 - 当前持仓 |

**凯利公式实现**：

```javascript
function kellyAllocation(funds, totalCash) {
    const available = totalCash * 0.8;  // 留20%现金
    let allocations = [];
    
    for (let f of funds) {
        if (f.score < 3.3) continue;  // 评分太低不买
        
        // 胜率 = 评分归一化
        let p = f.score / 5.0;
        // 赔率 = 近1年收益 / 最大回撤（近似）
        let b = Math.max(f.return1y, 0) / Math.max(f.maxDrawdown, 5);
        // 凯利比例
        let kelly = (p * b - (1 - p)) / b;
        kelly = Math.max(0, Math.min(kelly, 0.2));  // 限制 0%~20%
        
        // 建议金额 = 可用资金 × 凯利比例 × 评分加权
        let suggested = available * kelly;
        // 不超过日限额
        if (f.dayLimit > 0) suggested = Math.min(suggested, f.dayLimit);
        // 不超过总资金15%
        suggested = Math.min(suggested, totalCash * 0.15);
        
        allocations.push({fund: f, amount: suggested, kelly: kelly});
    }
    
    return allocations;
}
```

## 八、最终决策

| 问题 | 决策 | 理由 |
|------|------|------|
| 穿透估值维度 | **质量分子指标**（质量分内部新增估值子项，占比10%） | 改动最小，估值本身就是质量的一部分 |
| LLM 点评方式 | **B 方式**（管道输出 JSON，fund-scan Skill 读取后 LLM 解读） | 无需 API Key，免费，且符合"LLM 只做解读不做评分"原则 |
| 资金分配方案 | **自动计算**：从 `my_holdings` 汇总持仓总金额，预留20%现金，其余按评分加权分配 | 无需用户手动输入，持仓总金额自动反映可用资金 |
