## 4. 加仓 (建仓/加仓区分) 跑出了什么

**60 天里加仓 0 次**。

原因：
- LLM 5-27 才买入 013841，机器 5-28 看评分认为不需要加仓
- 6-01 之后持仓基本稳定，评分没有持续超阈值的"加仓信号"
- 加仓路径实现 OK (`compute_buy_amount` 函数已加)，但**没有触发场景**

**结论**：加仓区分已实现，但当前市场环境（4-7 月单边涨 + 评分稳定）不触发加仓。要触发需要：
- 持仓基金短期回调 -5% 但长期趋势向上
- 评分突然从 3.5 升到 4.5
- 同主题出现 2 只候选（但当前只买 1 只）

## 5. 关键发现

### 🚨 发现 1: LLM v2 的"软维度"过头了

v2 prompt 让 LLM 用 4 类软维度（同经理/规模/集中度/同主题）做否决。**但同主题/集中度这 2 项砍掉了收益冠军**。

**真实数据**:
- 机器 4-28 后 5 只持仓全是 tech/QDII/AI 主题
- 这 5 只最终贡献了 +19.27% 的绝大部分收益
- LLM 把 "tech 集中" 当风险 → 砍掉最赚钱的

### 🚨 发现 2: LLM v2 的 1 个正确否决

**6-22 否决 501226**:
- 规模 5.4 亿 (太小)
- 1 年 +38% (同类最弱)
- 7-11 浮盈仅 +0.62% (买 500 元几乎无贡献)

**LLM 这次否决真的省了**——500 元仓位虽然小，但 -0.62% 浮盈 vs 砍掉这笔可换成现金。

### 🚨 发现 3: "机器 + LLM veto" = 实际效果 = LLM 是负贡献

机器 5 维评分 + 进化算法 + 仓位控制**已经足够好**。LLM 的"软维度"否决**没有补强机器的盲点，反而砍掉了机器的最优解**。

---

## 6. 改进方向 (按收益预期排序)

### 🔥 方向 1: 缩小 LLM veto 范围 (建议)

v3 prompt 应该只让 LLM 否决**真正有问题的**:
- ✅ 规模 < 5 亿 (清盘风险)
- ✅ 经理刚变更 (公告)
- ✅ 单笔 > 25% 仓位 (硬约束)
- ❌ ~~同主题~~ (砍掉冠军)
- ❌ ~~持仓集中度~~ (机器已经在管)

**预期效果**: LLM 6-22 否决 501226 (对) + 不否决 013841/022184/024663 → 收益约 +18% (vs 当前 +10%)

### 方向 2: 加仓路径改造 (已完成, 待真实触发场景)

`compute_buy_amount` 已实现建仓/加仓区分。当前 60 天没触发是数据问题不是代码问题。

### 方向 3: 不接 LLM (干净解)

如果 LLM 接进来总是负贡献，**建议不接 LLM**——机器 + 进化算法已经比 90% 量化策略强。

但 LLM 仍有 2 个不可替代价值:
- 看公告 (经理离任/分红/规模变动)
- 报告可读性 (给投资人看的叙述)

## 8. 最终结论

| 维度 | 答案 |
|------|------|

- v3 prompt 应该只否决"规模/经理/单笔上限"3 项，**不要否决主题/集中度**
